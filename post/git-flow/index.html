Git Flow 分支策略

November 26, 2019

  • git

Git-flow分支策略相关整理

Git Flow

Git Flow是由Vincent Driessen基于Git这一版本控制系统所总结出来的一套可行的、能够满足很多项目开发需求（注意：不是大多数项目）的敏捷开发流程。

  • 分支模型图

Git Flow分支

  • master分支

master分支上是最近发布到生产环境的代码，最近发布的可用的release，仅可以从release分支和hotfix分支合并，不可随意修改

  • develop分支

develop分支是主开发分支，合并自feature，包含所有要发布到下一个Release的代码（一般在提测阶段使用，提测结束后发往release分支）

  • feature分支

develop分支是一个特性分支，用来开发一个新的功能，一旦开发完成，我们合并到develop分支进入下一个release

  • release分支

release为版本分支，发布一个新release的时候，我们基于develop分支创建一个release分支，完成release后，合并到master和develop分支(一般在提测完成后，release为灰度环境版本，当灰度完成后合并到master和develop)

  • hotfix分支

当在线上环境发现bug后，我们需要创建一个hotfix分支, 完成bug fix后，会合并回master和develop分支，所以hotfix的改动会进入下一个release。

基于Git Flow分支模型的简单实践(代码示例)

  • 特性分支
# 创建特性分支 feature-x
git checkout -b feature-x

# 完成特性分支 feature-x 合并到develop进行提测
git checkout develop
git merge --no-ff feature-x
git push origin develop

## -- 可选(删除feature特性分支)
git branch -d feature-x
git push origin --delete feature-x

# 提测完成，develop合并到release进行灰度
git checkout -b release-0.1.0 develop

# 灰度完成，release合并到master，并打上tag，发布线上
git checkout master
git merge --no-ff release-0.1.0
git push

git checkout develop
git merge --no-ff release-0.1.0
git push

git tag -a v0.1.0 master
git push --tags

## -- 可选（删除release分支）
git branch -d release-0.1.0
git push origin --delete release-0.1.0
  • hotfix
# 创建hotfix分支进行修复
git checkout -b hotfix-0.1.1 master

# 合并到develop分支进行提测
git checkout develop
git merge --no-ff hotfix-0.1.1
git push

# 提测完成后合并master，发布线上
git checkout master
git merge --no-ff hotfix-0.1.1
git push

git tag -a v0.1.1 master
git push --tags

## -- 可选（删除hotfix分支）
git branch -d hotfix-0.1.1
git push origin --delete hotfix-0.1.1

很多时候标准方案并不是最好的，选择合适的最重要
